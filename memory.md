Memory System
What It Is
The memory system is how Agent learns and remembers. Every conversation, every coding session, every agent task feeds into a persistent knowledge layer that grows smarter over time. The goal is simple: Agent should know you the way a long-time colleague does — your preferences, your projects, your patterns, your decisions — without you ever having to manually tell it what to remember.
Core Principles
Memory is invisible. The user should never have to think about memory management. Extraction, storage, retrieval, and consolidation all happen automatically. The only visible effect is that Agent gets better the more you use it.
Memory is layered. Not all memories are equal. Some are always relevant (core facts about the user), some are contextually relevant (retrieved when the topic comes up), and some are archived knowledge that only surfaces through deep search. The system manages these tiers automatically based on importance, recency, and relevance.
Memory evolves. Information changes. Preferences shift. Projects start and end. The memory system must handle contradictions gracefully — newer information supersedes older information, and the dreaming cycle actively resolves conflicts rather than letting stale data accumulate.
Memory is connected. Individual facts are useful, but relationships between facts are where real intelligence lives. A knowledge graph connecting memories through typed relations (supports, contradicts, depends on, part of) enables the system to reason about context rather than just pattern-match against isolated snippets.
Memory forgets. An infinite perfect memory is actually worse than a selective one. Unimportant, unused memories should naturally decay and eventually be pruned. The forgetting curve ensures the system stays lean and relevant rather than drowning in noise.
The Four Problems

1. Storage
   Everything lives in Postgres. Structured data, full-text search, and vector embeddings all in one database through pgvector. Each memory is a concise factual statement with metadata: category, importance score, embedding vector, timestamps, and relation links. The knowledge graph is just a relations table with typed edges. No external graph database, no Redis, no separate vector store. One database, one source of truth.
2. Extraction
   Every interaction is raw material. The extraction layer watches for high-signal moments: user corrections ("actually I prefer X"), explicit decisions ("let's go with Drizzle"), demonstrated patterns (solving the same type of problem repeatedly), and contextual facts (mentions of projects, people, preferences). A lightweight extraction runs after each conversation. A deeper extraction runs during the dream cycle. Both produce structured memory objects that get deduplicated against existing knowledge before storage.
3. Retrieval
   When a new session starts, the system assembles relevant context from memory. This is multi-strategy: semantic similarity via vector search finds conceptually related memories, importance weighting surfaces the most significant ones, recency biases toward current information, and graph traversal pulls in connected context. The retrieved memories are formatted into a natural system prompt that makes the model feel like it genuinely knows the user. Every memory used in a response gets its access count bumped, reinforcing its importance.
4. Consolidation
   The dreaming cycle runs periodically during idle time. It scans accumulated conversations, extracts new knowledge, merges duplicates, resolves contradictions, creates new relations, applies the forgetting curve, and prunes decayed memories. The result is a cleaner, more coherent memory store that improves session startup quality and reduces noise. Think of it as defragmenting a hard drive — the data is the same, but it's organized better so everything runs faster.
   How It Connects to Everything Else
   Memory feeds into every part of the platform. The chat UI gets personalized context injected at session start. The agent task system draws on memory to understand what matters to the user when prioritizing research or coding tasks. The coding harness gets preferences and patterns from past sessions. The dreaming cycle processes transcripts from all sources — chat, coding, agent runs — into unified knowledge. The artifact system can query memory to build personalized tools. Even the cost dashboard benefits, because smart model routing can be informed by what the system knows about query complexity from past patterns.
   Memory is the connective tissue. Without it, Agent is just another chat UI with an API key. With it, Agent is a system that understands its user and gets better every day.
